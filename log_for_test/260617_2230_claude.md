## 태스크 1: 오디오 분석 → 프롬프트 재료

### 현재 흐름
`audio.py` → `rms_mean, centroid_mean, flatness_mean, chroma_mode, tempo`
→ `music_analyzer.py` → `style_tags, color_hint, mood, avoid_tags`
→ `generator.py` 조립: `f"{mood}, {color_hint} tones, {style_tags}, {base_enhanced_prompt}"`

### 발견된 문제

**[문제1] 모션 동사 부재**
현재 style_tags는 형용사 중심(`"dramatic"`, `"cinematic"` 등). 비디오 확산 모델은 모션 동사가 있어야 실제 움직임이 생성됨. `bpm_feel`이 `"fast cuts"`까지는 생성하지만, `"camera zooms"`, `"particles burst"` 같은 동사형 구가 없음.

**[문제2] 에너지 곡선 형태 미반영**
`rms_std`로 "dramatic/steady" 구분은 되지만, 분포의 모양(급격한 폭발 vs 꾸준한 빌드업)은 잡지 못함. 단일 통계값으로는 에너지 아크를 표현할 수 없음.

**[문제3] 프롬프트 조립 순서 비최적**
현재 순서: `mood → color_hint → style_tags → base_enhanced_prompt`
비디오 모델에서 모션 기술어는 앞에 올수록 가중치가 높음. 색감과 분위기보다 모션이 먼저 와야 함.

### 권고 전략
- `bpm_feel` → 모션 동사 매핑 테이블 추가 (`"uptempo" → "fast swirling motion"` 등)
- Gemma 시스템 프롬프트에 `"use at least 2 motion verbs"` 조건 추가
- 조립 순서 변경: `모션 묘사 → 에너지/무드 → 스타일 태그 → base_prompt`

## 태스크 2: 프레임별 Beat CFG

### (a) LTX 시간 압축 공식 버그

LTX Causal VAE 구조:
- Latent frame 0 = video frame 0 (정확히 1프레임)
- Latent frame t (t≥1) = video frames `[1+8(t-1) : 1+8t]`의 pool, 중심 시각 = `(4.5 + 8*(t-1)) / fps`

**현재 코드** (`beat_cfg_denoiser.py`): `time_t = (8 * t) / peak_fps`

| t | 현재 공식 (fps=8) | 실제 중심시각 | 오프셋 |
|---|---|---|---|
| 0 | 0.0s | 0.0s | 0 |
| 1 | 1.0s | 0.5625s | −0.44s |
| 2 | 2.0s | 1.5625s | −0.44s |

BPM=120 (비트 간격 0.5s)에서 CFG 부스트가 인접 비트에 오배정될 수 있음.

**수정 필요**: `time_t = 0.0 if t == 0 else (4.5 + 8 * (t - 1)) / peak_fps`

### (b) 마스터 비디오 ↔ 전체 영상 타임라인 불일치 ← 핵심 설계 문제

현재 `beat_timestamps`는 **전체 오디오 기준** (예: 0~180초). 그러나 마스터 비디오는 **~16초짜리 클립**으로 생성 후 가변 배속 + 반복 조립됨. 생성 시점에 t=3초에 CFG 부스트를 줬더라도, 완성 영상에서 그 구간이 어느 타임코드에 어떤 배속으로 재생될지는 생성 이후에 결정됨.

**결론**: Per-Frame Beat CFG의 "비트 동기 콘텐츠 변화" 주장이 이 불일치로 인해 논문에서 성립하지 않음.

**권고 재프레임**: 기여를 "비트 위치에서 로컬 품질(선명도·프롬프트 충실도) 강화"로 재서술.

논문 표현 초안:
> *"The per-frame CFG modulation enhances local detail fidelity at beat-aligned latent positions during master clip synthesis. Precise temporal mapping to the final assembled video is an open problem due to speed warping."*

### (c) 모델 비종속성 주장 범위 문제

`loop_samplers.py`, `beat_cfg_denoiser.py`가 `ltx_core`, `ltx_pipelines` 내부 클래스를 직접 상속·임포트함. "어떤 비디오 생성 모델도 끼워 쓸 수 있다"는 현재 주장은 구현과 불일치.

**권고**: "스케줄링·어셈블리 파이프라인은 모델 비종속. Latent-space 연산(루프 합성, Per-Frame CFG)은 현재 LTX-Video 기반으로 시연하나, flow-matching 모델 일반으로 이식 가능."

## 태스크 3: 루프 시 운동 방향 유지 문제

### 근본 원인

`loop_samplers.py`의 `cyclic_magnetic_steering_denoising_loop`:
- x0 tail 프레임을 head 프레임으로 **SLERP 블렌딩** → **외형(appearance)만 맞추고 모션 벡터(temporal derivative)는 무시**
- velocity stitching (`delta_head` 전파)이 있지만, head의 모션 방향을 tail에 이식하는 과정에서 **전환 구간이 생성**

결과: 루프 경계 직전 프레임들이 콘텐츠는 head처럼 보이면서 기존 모션 방향으로 "당겨지는" 효과 → 역방향 재생처럼 느껴짐.

### 전략 (우선순위 순)

**즉시**: 마스터 비디오 프롬프트에 순환 모션 강제
- 추가: `"pulsing"`, `"expanding and contracting"`, `"rotating vortex"`, `"oscillating waves"`
- 억제: `"camera pan"`, `"moving forward"` 등 선형 이동 유도 표현
- 근거: 선형 이동 콘텐츠는 원리상 완전 루프 불가. 회전/맥동/진동 콘텐츠는 자연적으로 루프됨.

**단기**: 루프 경계점 자동 탐지
- 각 프레임 쌍 (i, j)에 대해 외형 유사도 + 모션 방향 유사도를 동시에 계산
- 가장 유사한 쌍을 루프 경계로 설정하여 클립 트리밍 → FFmpeg crossfade 연결

**연구**: FLF2V 양단 앵커 (Phase 4 "2nd Latent Anchor" 보류 항목 재개)
- 첫 프레임 = 마지막 프레임 조건으로 생성 → 가장 완전한 해결책

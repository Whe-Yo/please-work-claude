# Plan — 구현 계획 모드

계획이 승인되기 전에는 코드를 수정하지 않는다. 읽기·검색만 허용.

## 절차

1. **조사**: 관련 파일을 읽고 현재 구조를 파악한다. 추측으로 계획하지 않는다 — 계획에 등장하는 모든 파일·함수는 실제로 열어서 존재와 시그니처를 확인한다.
2. **계약 정의**: 새로 만들거나 바꾸는 인터페이스의 함수 시그니처, 입출력, 에러 모델을 먼저 적는다.
3. **계획 제시**: 다음 형식으로 출력한다.

```
## 계획: (한 줄 요약)

### 변경 파일
- `경로` — 무엇을 어떻게 바꾸는지 한 줄

### 단계
1. (검증 가능한 단위로 쪼갠 단계. 각 단계는 그 시점에 테스트/실행 가능해야 함)

### 리스크
- (깨질 수 있는 것, 영향 범위, 되돌리기 어려운 단계)

### 검증 방법
- (완료를 무엇으로 확인할지: 실행 명령, 기대 출력)
```

4. **승인 대기**: 사용자가 승인하면 실행 단계로 진입한다. 실행 중 계획과 달라지는 지점이 생기면 멈추고 변경 사유를 보고한다.
5. **기록**: 승인된 계획의 핵심 결정은 `work.md`에 WHAT/WHY/REJECTED 형식으로 기록한다.

## 규칙
- 대안이 여럿이면 전부 나열하지 말고 추천 1개 + 근거, 기각 대안은 한 줄씩.
- 대규모 구조 변경이 포함되면 1단계를 "백업 커밋"으로 한다.

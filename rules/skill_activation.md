# 스킬·기능 발동 기준표

각 기능이 **언제** 켜지는가를 명시한다. 목적: "언급할 때만/잊혀짐"을 없애되, **자동 기준을 못 박을 수 있는 것**과 **상황 판단이 필요한 것**을 구분한다.

## 유형
- **하드(훅)**: Claude Code 훅으로 결정론적 발동. 에이전트가 못 잊음.
- **소프트(넛지)**: 훅이 매 턴 상태 보고 조건 충족 시 환기(process-status). 실행은 에이전트.
- **판단**: 상황 인식이 필요해 자동 기준을 못 박음 → 지시층(규칙)으로 두고 에이전트가 판단. ("할 수 있는건"의 경계 밖.)

## 표

| 기능 | 발동 기준 | 유형 | 상태 |
|---|---|---|---|
| **boost** | 세션 시작(startup/resume) | 하드(SessionStart 훅) | ✅ |
| **antithesis** | 미검토 편집 `pending ≥ 3`(작업 묶음당 1회, Task 실행 시 리셋) | 하드(Stop 게이트) + 소프트(넛지) | ✅ |
| **clemini 하달** | 마지막 agy 하달 이후 쌓인 작업(편집+조사 증분) `≥ 2` — 하달하면 스냅샷 갱신으로 방출(좀비 없음). 사용자 지시 260702: 팍팍·병렬·백그라운드, 결과 미반영 OK | 소프트(넛지) — **슬롯 밖 독립·최우선 노출** | ✅ 넛지 |
| **worklog(RPW)** | RPW 없음 & 편집 `≥ 2` / 작업 단위 완료 | 소프트(넛지) | ✅ 넛지 (재프레임=작업완료 마커는 백로그) |
| **Datavault** | 편집 `≥ 5` & 노트 0 (성역·비-작업 프로젝트 제외) | 소프트(넛지) | ✅ (컨텍스트 게이트는 백로그) |
| **verify** | 수정 후(pending≥2) — 동작 확인했나 | 소프트 자문(process-status) | ✅ |
| **paper** | `.tex`/`.bib` 편집 감지 | 소프트 자문(process-status) | ✅ |
| **review** | (verify와 중복 — 제외) | — | ➖ |
| **manage** | (확인 자체가 토큰 낭비 — 제외) | — | ➖ |
| **setup** | 저장소 최초 적용 1회 | 하드(1회성) | 수동 |
| **plan** | 파일 3개↑ 수정·아키텍처 변경·30분↑ 예상 | **판단**(착수 *전* 예측이라 자동 감지 곤란) | 규칙만 |
| **peer** | 고난도 문제 2회 실패 | **판단** | 규칙만 |
| **resource** | 무거운 작업(실험·서버·학습) 착수 전 | **판단** | 규칙만 |
| **feedback** | 버그·마찰·개선점 발견 시 | **판단(능동 의무)** — 사용자 지시 없이 즉시(§6) | 규칙만 |

## 원칙
- **하드 > 소프트 > 판단** 순으로, 가능한 한 위 단계로 끌어올린다(잊혀짐 방지). 단 **판단형을 억지로 임계치화하면 오발동**(안티테제 문제 B/G와 동류) → 무리하지 않는다.
- 소프트 넛지/자문은 **우선순위 상위 최대 2개**만, **조건 충족 시에만** 출력(clean이면 침묵) — process-status와 일치(안티테제 1-C).
- verify·paper는 자문에 편입(✅). review(verify와 중복)·manage(확인 비용>이득)는 **제외 결정** — 무리한 자동화는 과잉(안티테제 경고).

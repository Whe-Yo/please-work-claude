# Roadmap — 보류된 설계 (검증 후 도입)

이 프로젝트는 "더하지 말고 깎아라"를 원칙으로 한다. 아래 기능은 가치 가설은 있으나 **선행 기능이 실제로 부족함을 입증한 뒤에** 도입한다. 검증 없이 미리 만들면 과잉설계가 된다.

---

## 1. 리소스 예약 원장 (Resource Reservation Ledger)

**선행 조건**: `resource` 스킬(라이브 조회)을 실사용해서, "현재 실측만으로는 부족하다 — 남이 *곧* 무엇을 돌릴지 몰라 충돌이 반복된다"가 확인될 것.

**왜 보류**: `resource`의 라이브 조회(nvidia-smi/ps)가 사용자 통증의 대부분("지금 뭐가 도나")을 이미 해결한다. 예약 원장은 "미래 의도" 정보만 추가하는 부가물이라, 라이브 조회로 부족함이 드러나기 전엔 비용이 가치를 넘는다.

**설계 (도입 시)**:
- **저장은 의도, 실측은 조회 — 분리 원칙 유지.** 원장은 "실행 중" 저장소가 아니라 "예약/의도" 원장이다. "지금 뭐가 도나"는 항상 라이브 조회로 답한다.
- **한 파일에 섹션 나누기 금지 → 프로젝트별 개별 파일.** `~/.agents/reservations/{프로젝트}.md`(또는 `.json`). 이유: 단일 파일 공유는 다중 에이전트 동시 쓰기 레이스를 부른다. 프로젝트별 파일은 쓰기 충돌이 0이고, "남의 파일 안 건드림"으로 스코프 쓰기 권한이 자연 강제된다.
- **착수 절차**: (1) `resource` 라이브 조회로 현재 실측 + (2) `reservations/` 전체를 읽어 타 프로젝트의 선언된 의도 확인 → 충돌 판단.
- **drift 방지**: 각 예약에 만료 시각(TTL) 필수. 만료 지난 예약은 무시. 가능하면 PID 생존과 대조해 좀비 예약 제거.
- **여전히 소프트**: Claude가 예약 기록을 깜빡하면 원장이 비게 된다. 강제하려면 강제 층(Claude Code Hook: 무거운 명령 실행 전 예약 기록 확인)이 필요.

**의도적 제외(원장 도입 후에도)**: 자동 디스패치·스케줄러·데몬. 큐→실행 전이를 자동화하면 데몬이 필요하고, 그 순간 경량 원칙을 벗어난다. 전이는 사람/Claude 호출로 둔다.

## 2. feedback: 브랜치/PR 모델 전환

**선행 조건**: feedback의 main 직접 커밋에서 push 경쟁·동시성 문제가 실제로 반복 확인될 것.

**왜 보류**: 프로토타입 단계는 트래픽이 낮아 main 직접 커밋(가드 포함: 파일만 add, rebase 재시도 2회, 범위·비밀 점검)으로 충분. 안티테제는 단일 공유 main의 위험(경쟁·오염)을 지적했고 `feedback/{agent}` 브랜치+PR을 권고했으나, 브랜치 관리는 프로토타입에 과한 복잡도.

**설계 (도입 시)**: feedback이 `log_for_test/` md를 main이 아닌 `feedback/{agent}` 브랜치에 커밋·푸시하고, 유지보수자가 병합. main 직접 쓰기 제거 → 경쟁·오염 원천 차단.

## 기록 규칙
- 보류 기능이 검증되어 도입되면 이 항목을 제거하고 해당 스킬/문서로 옮긴다.
- 기각된(도입 안 하기로 한) 설계는 `agents_chat.md`의 결정 로그 또는 커밋 메시지에 사유와 함께 남긴다.
